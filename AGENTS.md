# Repository Instructions

## Environment
- Backend Python dependencies are installed in the Conda environment `alarino`.
- For backend Python commands that rely on installed packages, use `conda run -n alarino ...` or activate `alarino` first.
- Do not assume the active shell is already using `alarino`; verify before running backend tests or import checks.

## Database migrations (Alembic / Flask-Migrate)

**Default workflow: generate-and-edit.** When a schema change needs a migration, edit the model first, then run `flask db migrate -m "..."` to autogenerate the revision file, then edit the produced file before committing. Don't hand-roll from scratch — autogenerate catches model/migration drift, gets boilerplate right, and surfaces surprises early. Hand-roll only for pure data-shaping migrations that have no schema diff.

**Always edit the autogenerated file** for these things autogen doesn't handle:
- Column renames are produced as `drop_column` + `add_column`, which loses data. Replace with `op.alter_column(name, new_column_name=...)`.
- Add backfills (UPDATE / INSERT) and any data-shape transforms.
- Add preflight checks for destructive operations — fail loud with offending row IDs rather than corrupt data silently.
- Add sidecar audit tables before any DELETE / DROP COLUMN so removed data is recoverable.
- Use opt-in env-var escape hatches (e.g. `ALARINO_FORCE_DOWNGRADE_DATA_LOSS=1`) on irreversible downgrades.
- Rename autogenerated constraints to project conventions: `ck_*` for CHECK, `unique_*` for UNIQUE, `idx_*` for indexes, `fk_*` for foreign keys, with explicit names rather than DB defaults.

**Verify forward + backward** before committing: `flask db upgrade head && flask db downgrade -1 && flask db upgrade head` against a fresh local DB.

**Run the migration against a prod-shape DB before merging.** Autogenerate-and-edit catches schema/model drift, but it doesn't catch issues that only surface against real data — e.g., a unique constraint that prod data already violates, or a preflight that fires on existing rows. Use a Neon branch / staging copy / `pg_dump` snapshot.
